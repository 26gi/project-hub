## セットアップ手順（ローカルで動かす方法）

### 前提条件

- Docker Desktop がインストールされていること
- ポート `5173`（フロント）・`8000`（バックエンド）・`5433`（DB）が空いていること

### 手順

```bash
# 1. リポジトリをクローン
git clone <repository-url>
cd project-hub

# 2. フロントエンドの依存パッケージをインストール
cd frontend
npm create vite@latest . -- --template react-ts
npm install
cd ..

# 3. 起動
docker compose up --build
```

### アクセス先

| URL                        | 内容                         |
| -------------------------- | ---------------------------- |
| http://localhost:5173      | フロントエンド（案件ボード） |
| http://localhost:8000/docs | FastAPI Swagger UI           |

## 技術スタック

| レイヤー       | 技術                           |
| -------------- | ------------------------------ |
| フロントエンド | React / TypeScript / Vite      |
| バックエンド   | FastAPI / Python 3.11          |
| データベース   | PostgreSQL 15                  |
| ORM            | SQLAlchemy（非同期 / asyncpg） |
| インフラ       | Docker / Docker Compose        |

## 技術スタックを選んだ理由

### React

Next.jsとの比較で決定

- 経験少ないながらも一応触ったことのある技術のため
- 今後機能を拡張していく場合にコンポーネント単位で部品を管理できるため
- 要件は「画面の操作性」と「業務効率」でありSEOが不要なため

### Python

- 普段実務で使用しているC#で一からWebシステムを作成したことがない
  →実務、自習経験のあるPythonを使用

### FastAPI

Flaskとの比較で決定

- Pythonフレームワーク内で比較したときに最もパフォーマンスが高いため
- バリデーションが標準で備わっているためコード量を削減できるため

### PostgreSQL

- 直近で一番触っているDBのため

### ORM(SQLAlchemy)

ストアドとの比較で決定

- 一覧・登録・更新・削除のシンプルなCRUDのみのため
- 大量のデータを扱うわけではないため
- モデル定義をコード管理可能なため
  →変更履歴をGitで追跡可能なため保守性⚪︎

### Docker

- 環境構築時に使い慣れているため
- 複数人での開発に切り替わっても全員が同一環境で開発可能であるため

## 必須要件以外に実装した機能と、追加した理由

| 機能                       | 理由                                                                 |
| -------------------------- | -------------------------------------------------------------------- |
| 売上見込み算出・表示       | 別資料で集計、保持しなくてもこのシステムで可視化できるようにするため |
| 提案資料・見積書管理機能   | 資料の共有がこのシステムのみで可能になり、案件への属人化を防げるため |
| 企業選択モーダル・絞り込み | 同じ顧客で複数案件受注するケースに対応するため                       |

## 実装しなかったが「あると良い」と思った機能と、その理由

| 機能 | 理由 |
| ----------------------- | -------------------------------------------------------------------- | |
| タスク管理機能 | 案件が始まればその中でのタスク切り分けが必要になるため |
| ログイン機能 | ユーザごとのタスク管理などが可能になるため |
| 承認フロー機能 | 実際に次のフェーズに進めるときに上長の承認を得るフローが考えられるため |
| カンバンボード機能 | 操作を直感的にできるため(ドラッグ&ドロップなど) |
| ダッシュボード機能 | 各案件にどのくらいリソースが割かれているか各ユーザにどのくらい負荷がかかっているかを可視化できるため |
| 顧客管理機能 | 過去の取引を参考にできるため(分析やリピート時に役立ちそう) |
| 案件参画者登録機能 | 参画状況を確認できると他案件のタスク振り分け時に負荷を分散できるため |
| 見積書作成補助機能 | 誰でも見積もりが作成可能になることで営業活動の属人化を防ぐため |

## 時間があればやりたかったこと

- バリデーション(文字数制限や期待しない入力を未然に防ぐ実装を意識していない)
- デザイン(BacklogやJiraなど既存システムのように洗練されたデザインにしたかった)
- もっと保守性を考えた設計、実装(とりあえず形になることを優先)
